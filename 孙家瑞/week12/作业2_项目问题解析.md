# 前后端分离与历史对话存储

## 什么是前后端分离？

前后端分离是一种软件架构模式，将前端（用户界面）和后端（业务逻辑）拆分为独立的模块，通过 HTTP API 进行通信。

在本项目中：

- **后端（端口 8000）**：`main_server.py` 基于 FastAPI 实现，提供 REST API
  - `routers/` - 用户、聊天、数据、股票等模块化路由
  - `api/autostock.py` - 股票相关 API 挂载在 `/stock` 路径下

- **前端**：Streamlit 多页面应用（`demo/streamlit_demo.py`）
  - 通过 `st.navigation()` 实现页面路由
  - 页面分组：用户中心、股票中心、数据中心、工具中心、智能问答

- **MCP 服务器（端口 8900）**：`main_mcp.py` 基于 FastMCP，提供工具能力
  - 集成天气、地址解析、新闻、汇率等 MCP 工具

前端通过 **HTTP REST API** 与后端通信，聊天功能使用 **Server-Sent Events (SSE)** 流式返回结果。

## 历史对话如何存储？

使用 **SQLite** 数据库存储对话历史：

**数据库位置**：`assert/conversations.db`

**两张核心表（`models/orm.py`）**：

| 表名 | 字段 | 作用 |
|------|------|------|
| `chat_session` | id, user_id, session_id, title, start_time, feedback | 会话元数据（标题、反馈） |
| `chat_message` | id, chat_id, role, content, generated_sql, generated_code, create_time | 每条消息内容 |

**角色类型**：system / user / assistant

## 如何将历史对话作为大模型的下一次输入？

项目采用 **两层记忆机制**：

### 1. OpenAI Agent SDK 的 `AdvancedSQLiteSession`（主要机制）

```python
# services/chat.py
session = AdvancedSQLiteSession(
    session_id=session_id,
    db_path="./assert/conversations.db",
    create_tables=True
)
```

`AdvancedSQLiteSession` 是 SDK 内置的对话记忆组件，会自动：
- 每次对话时，从数据库查询该 `session_id` 的所有历史消息
- 将历史消息作为上下文，连同新的用户输入一起送给 LLM

### 2. 流程简述

```python
# 1. 创建或恢复会话
if session_id:
    record = session.query(ChatSessionTable)...
    if not record:
        init_chat_session(user_name, content, session_id, task)

# 2. 将用户消息存入数据库
append_message2db(session_id, "user", content)

# 3. 获取系统提示（包含任务说明）
instructions = get_init_message(task)

# 4. 创建 OpenAI 客户端和 MCP 服务器
external_client = AsyncOpenAI(api_key=..., base_url=...)
mcp_server = MCPServerSse(name="SSE Python Server", params={"url": "http://localhost:8900/sse"})

# 5. 创建带历史记忆的 session，运行 Agent
session = AdvancedSQLiteSession(session_id=session_id, db_path="./assert/conversations.db")
result = Runner.run_streamed(agent, input=content, session=session)
```

### 3. 系统提示注入

系统提示通过 Jinja2 模板（`templates/chat_start_system_prompt.jinjia2`）生成，在会话初始化时作为 `instructions` 传入 Agent，确保每次新会话都有正确的任务上下文。

### 4. 会话管理的额外表

`assert/sever.db` 中还有 `ChatSessionTable` 和 `ChatMessageTable`，用于应用层面的功能（如会话列表展示、用户反馈），独立于 Agent SDK 的记忆机制。
