
# 在客服工作台项目中，作为后端开发和算法开发的角色，我们需要做以下事项

## 1.设计数据库

```
需设计多表结构适配 FAQ 全流程管理，核心包含类目表（支持无限级分类，存储类目 ID、名称、父级 ID、层级等）、FAQ 主表（存储 FAQID、标题、答案、生效时间、类目关联 ID、答案类型、标签、环境状态 （测试/正式）等）、相似问法表（关联主 FAQID，存储最多 200 条相似问法）、关联问题表（存储 FAQ 间的关联推荐关系）；
同时预留批量操作、环境同步相关字段，区分测试/正式环境的 FAQ 数据状态。
```

## 2.开发接口

```
开发类目管理接口：支持多级类目增删改查、平级/下级创建、类目检索、数据量刷新，限制含下级/关联 FAQ 的类目不可删除。
开发 FAQ 全生命周期管理接口：实现新建、编辑、类目转移、失效、删除，以及批量导入、导出；做双环境隔离开发，测试环境操作需经发布接口同步至正式环境，保障正式环境数据不可直接增删改。
开发搜索与匹配落地接口：接收算法层的相似度匹配结果，关联查询对应 FAQ 的答案、生效状态等，返回给前端并保障数据时效性。
```

## 3.使用BERT模型

```
核心使用 BERT 模型，暂无需使用大模型。BERT 足以满足 FAQ 短文本的语义相似度匹配需求，大模型的能力冗余且会增加部署成本，现有场景下暂无引入必要。
BERT 模型的使用方式：将用户提问、FAQ 标准标题、所有相似问法统一做文本预处理（分词、归一化）后，输入 BERT 模型进行语义向量化编码，提取文本的深层语义特征向量；
通过计算用户提问向量与历史 FAQ（标题+相似问法）向量间的余弦相似度，排序后选取相似度最高的结果，匹配对应 FAQ 的回答返回。
```
